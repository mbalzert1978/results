## Recommended order & rationale

All five are independent (no hard `Depends on`), so they can be done in any
order or in parallel. Recommended sequence reflects leverage:

1. **001** — the only real defect: a user with a value typed `Option[T]` gets a
   `mypy` error calling `.is_some()`/`.is_some_and()`. Highest leverage, additive.
2. **002** — locks the green baseline into CI so the changes below are gated
   automatically. No code risk.
3. **003** — mechanical internal rename; pure tech-debt cleanup, zero behavior
   change.
4. **004** — public breaking change (pre-1.0, cheap); writes a superseding ADR.
5. **005** — curated Rust-parity addition; nice-to-have, not a fix.

## Findings considered and rejected

- **Hash collision across variants** (`Ok(1)`, `Err(1)`, `Some(1)` all hash to
  `hash(1) * 41`): not worth doing. Harmless — the typed `__eq__` on each variant
  disambiguates, and mixed-type sets/dicts are rare. No correctness impact.
- **Shared value-carrier base/mixin** to de-duplicate `__eq__`/`__hash__`/
  `__repr__` across `Ok`/`Err`/`Some`/`Null` (the "deep rung" of the naming
  finding): rejected. `docs/decisions/issue-13-split-option-module.md` §E2 already
  rejected a shared private base as YAGNI; the duplication is four short, stable
  methods. Only the cheap rung (rename, plan 003) is worth doing now. Revisit if
  the `* 41` salt or equality rule ever starts changing.
- **Add `Result.some`/`Result.none` for symmetry** (alternative to deleting
  `Option.some`/`none`): rejected. `Result.ok()`/`err()` already exist as
  conversions to `Option`; constructor classmethods would name-collide and add
  surface. Plan 004 deletes instead.
- **`ruff format --check` as a CI gate now:** deferred, not rejected. One file
  (`tests/results/test_public_api.py`) is currently unformatted, so enforcing it
  would make CI red on day one. Plan 002 omits it and documents the follow-up
  (run `uv run ruff format`, commit, then add the gate).
